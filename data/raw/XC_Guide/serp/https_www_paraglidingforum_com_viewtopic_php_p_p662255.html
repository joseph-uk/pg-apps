Long story short: You need to disable the output Vario in the Output settings of GXAirCom (see attached screenshot)

Maybe this helps someone at any rate...

Manfred

Posted: Tue May 30, 2023 22:40 UTC    Post subject: RE: Android Apps: XC Guide & PG Race
Apologies Manfred, I was just preparing an answer to this question.

XC Guide will forward these sentences, once per second, to any other App connected to it via the localhost TCP connection, on the 'FLARM emulation port' 10110 :-

For a valid 3D GPS fix it will create and send:
$GPGGA...
$GPRMC...

From a connected XC-Tracer it will forward:
$XCTRC...
$LXWP0...

From a connected GxAirCom it will forward:
$LK8EX1....

For each Pilot List entry it will create and send:
$PFLAA,0,...

For every ADSB aircraft detected it will create and send:
$PFLAA,0,...

------

The sentence $LK8EX1 from a GxAirCom contains a pressure reading.

The format is
$LK8EX1,pressure,altitude m,vario cm/s,temperature C,battery V,*checksum

So if the connected App (e.g. xctrack) has another live pressure source, the two values will probably clash, especially if one is invalid. So only use one source for the pressure reading.

But you deduced this already, excellent.

indy

For info, there are a few niggly bugs, affecting the PG-Race and XC Guide Android Apps that, despite a good deal of investigation so far, Simon and I have been unable to resolve;

1) Task and IGC file upload to the PG-Race server. The Apache server began returning a "403 forbidden" failure, to file POST upload attempts, a few weeks ago. As a work-around I have released V122 of the PG-Race App which now opens the Web App at https://pg-race.aero/mobile/ for these two particular actions, (only if the server upload attempt fails). The Web App continues to work normally as do the other PG-Race App features.

2) https://XCView.net messages from XC Guide into the Telegram group using the bot, have stopped working. The App still reports 'message sent', but this only means it was delivered into the XCView.net server, it can't tell if the message was delivered into the group. It seems to be caused by some kind of new ISP <-> Telegram bot security rule, but we don't think it will have a quick resolution. XC Guide live-tracking into and out of XCView.net continues to work fine. The future fix may be for XC Guide to have its own bot just for messages, and have the App interact with Telegram directly.

3) XC Guide is forced to close by Android after a few hours running in the background, with no user interaction. This bug can be reproduced on a Google Pixel 6, running Android 13. After about 3 hours of the App starting normally, then being hidden behind other Apps, the OS closes the App with an unsolicited 'onDestroy' call-back. The App isn't crashing, it's being shut down. But not by the user. This is despite the persistent foreground notification being present, and background battery usage being unrestricted. It might affect other models of phone too? It doesn't happen when the App is in the foreground (visible) or even in split-screen mode.

Make sure the MipBip MAC address field is empty to start a new scan - the App will find this automatically.

Once it is connected via BLE, the App can use that pressure reading for flight computer barometric altitude, FL flight level airspace determination, IGC pressure records and vario sounds etc.

Posted: Thu Apr 04, 2024 8:35 UTC    Post subject: XC Guide on Onyx Boox with XC Tracer
Installing XC Guide on e-reader.
Onyx Boox Poke 2 and XC-Tracer Mini II GPS.
As Onyx hide some of the developer tools, I had to download another app -
Android Dev Launcher, which opens up the developer sections of Android (version 9.0).
Location is in Apps> App Management (menu item on top right) > Location setting.
The default setting is off, and there may not be a switch to change it (hidden by Onyx).
Once switched on then App-level permissions>Advanced scanning>Scanning needs to have
Bluetooth Scanning switched on.
Also in App Management, "Enable Google Play" needs to be on, otherwise no map is visible.

The XC-Tracer needs to have protocol set to XCTRACER, in order pass GPS position to XC Guide.
In XC Guide set Connected devices to "Use BLE" and XC Tracer to "GPS from XC Tracer".

Regards, Geoff

Another pilot at Rutland Airsports had a similar problem with an Onyx Boox, but now reports it working once the location issue was resolved.

The latest update V570 of XC Guide is now available in Play Store.
It has some important bug fixes and several new features;

Urgent bugfix: Problem with latest Google Map renderer causing low memory and crash after ~2 hours

SafeSky tracking added.
You can track yourself into SafeSky as well as see all traffic 100km around you.
Individual SafeSky pilots can be named as buddies and thus appear in your Pilot List.

Flight Computer Units (Metric / Imperial / Nautical) can now all be set individually, for altitude, distance, speed, climb rate, temperature and volume.

New 'generic' BLE vario e.g. for UltraBip

Routes can now be loaded from flyxc.app URL or QR code

RadarBox USB dongle is now supported to see ADS-B traffic

Long-press on Flight Computer altitude box to set GPS correction for 0 AGL.
If using barometric altitude, you can enter QNH in hPa (mb) for calibration

Excluded airspace can now be re-included individually by name or by whole class.

FANET messages can now be read aloud automatically.
The language used is based on whichever user guide you last opened.

NOTAM activations / deactivations can now be received by FANET broadcast message and are immediately drawn on the map. Airspace Proximity warnings are also re-calculated and alerted immediately.

When connected to your XC-Tracer Maxx2 via Bluetooth BLE, the IGC header details (pilot name, tandem passenger name, glider type etc.) can now be set in the Maxx2 from XC Guide or copied from the Maxx2 to the App. XC Guide can also now set the Maxx2 message type sent from the vario, e.g. parameter XCTRACER / LK8EX1 / LXWP0 / LXWPW. You can also download the last 5 IGC files from the Maxx2 to XC Guide and upload new firmware. This avoids the need to connect the Maxx2 to a PC which used to be necessary for these things.

The five-button Bluetooth BT006 is now supported.
Each button can be set to invoke any of the pre-set list of App functions.
Just like an iTag or the Fn custom button (or Air3 F1 & F2).

Version 570 APK is also available for sideload at:
https://pg-race.aero/xcguide/

The user guide is updated in the App and at:
https://pg-race.aero/xcguide/XCGuidehelp_EN.html
The non-English user guides need to be updated, however.
If any pilots, fluent in other languages, wish to contribute a technical translation, that would be great

I will provide more detailed posts on each new feature separately.

Posted: Wed May 29, 2024 17:45 UTC    Post subject: Spoken FANET messages
A few pilots asked for this feature.
It might be a bit of a gimmick but it may also be useful, we'll see..

For this feature to work:

1) The receiver needs to have a text-to-speech engine installed. They are free on Play Store.
For example Speech Recognition and Synthesis from Google.

2) The receiver needs to select Speak FANET messages in FANET & FLARM settings under Connected Devices.

3) The receiver needs to open the help guide in their preferred language.
XC Guide assumes that is your language, until you open a different one.

4) The sender needs to send messages, either pre-set or typed in, in the same language the receiver is set up for, otherwise the results are quite funny.

This feature is still not 100% finished but works, mostly.

When connected to a Maxx2 via BLE, XC Guide can copy the IGC header details (pilot name, tandem passenger name, glider type etc.) from the App settings into the Maxx2 or copy them from the Maxx2 to the App.

XC Guide can also now set the Maxx2 message type sent from the vario, e.g. parameter XCTRACER / LK8EX1 / LXWP0 / LXWPW.
This over-writes what was set in the text file e.g.
# supported protocols are None, XCTRACER, LK8EX1, LXWP0 or LXWPW
StringToSend=XCTRACER

You can also download the last 5 IGC files from the Maxx2 to XC Guide and upload new firmware.

The new firmware .iap files are not currently available via automatic download, so they need to be manually dropped into the indysoft.xc_guide/files/Feeds folder.

The airspace update function is not yet available from this menu. I am still working with Koni, continuing to test & perfect the Maxx2 upload / download protocols into XC Guide.

This feature of configuring your Maxx2 via an App, avoids the need to take a PC on your flying holidays, which used to be necessary for USB memory-type varios.

Posted: Wed May 29, 2024 21:50 UTC    Post subject: SafeSky via XC Guide
SafeSky, and the lightweight mobile ADS-L protocol using UDP, are potentially important developments for Electronic Conspicuity and situational awareness.

XC Guide can now track you into SafeSky as well as show you SafeSky traffic 100km around you. This traffic is an aggregation from numerous sources, and could be ADS-B, gliders, OGN, GA aircraft or UAVs etc.

Firstly you need to read and accept the terms and conditions of the service, provide an email address and a name. You also need to specify what aircraft type you are. These can be changed at any time. The App updates your registration automatically.

Individual SafeSky pilots can be named as buddies and will appear as regular trackers in your Pilot List.

Note that SafeSky buddies need to be within 100km AND within the altitude range filter that you set, or they won't be reported. This might change in future. I have asked for the API to be extended to allow query of buddies' locations anywhere.

SafeSky traffic reporting through the API is transient - it disappears from the App feed about 30s after the end of active tracking or loss of signal. However if a SafeSky pilot is in your buddy list, their marker's last known location will persist on the XC Guide map and in the Pilot List (until you edit the buddy list or the App is re-started).

When tracking into SafeSky you will appear here;
https://live.safesky.app/

More information on the service here;
https://www.safesky.app/en

SafeSky is particularly interesting as it may, in future, allow us free flyers access to EASA U-space;
https://www.easa.europa.eu/en/document-library/general-pub(shrinked URL)-will-enable-safe-integration

Posted: Wed May 29, 2024 22:18 UTC    Post subject: NOTAM activation / deactivation via FANET broadcast
This feature was a particular request from a club and the use case is currently subject of debate. However it potentially has use in other clubs and scenarios.

The way it works is that the specific named airspace Zone or ATZ to be activated / deactivated, by this NOTAM delivery method, already resides in the recipients' XC Guide airspace files.

In airspace settings, XC Guide already has a list of unique named zones / ATZs to ignore, and thus not plot on the map. The FANET message simply adds / removes named zones from this list, causing the maps to immediately be re-drawn, and the audible airspace proximity / penetration warnings to be offered / withdrawn straight away.

The sender (someone with a FANET message transmitter, authorised by the club and in contact with the authority) broadcasts a FANET message in the simple format shown below;

NOTAM ACTIVATE:DGA1
NOTAM DEACTIVATE:DGA1

In this example the airspace is called DGA1 in the OpenAir format file.

All pilots within FANET range (~25km) using XC Guide, will have their airspace maps updated automatically, as in the video example attached.

For yet another way to track ADSB traffic XC Guide now supports the USB dongle RadarBox.

The App ADSB Receiver from Avare, is also required;
https://play.google.com/store/apps/details?id=bs.Avare.ADSB
It should be set to localhost GDL90 mode - see screenshot of settings.

In Connected Devices settings, you will need to select RadarBox USB & GDL90 from Avare App and the App does the rest.

XC Guide will listen for GDL90 packets from this App, in a similar way to a SkyEcho2, and plot the traffic on the map.

Note that the USB dongle is power hungry and will flatten a phone quickly so use a USB OTG power splitter with extra power bank, as shown.

Using https://flyxc.app with Pop-up racing https://PG-Race.aero

flyxc.app is an excellent and very popular xc planning tool and is being actively developed.

The task QR codes from flyxc.app are now supported by PG-Race, pop-up racing and XC Guide.

There is a specific flyxc.app URL import option in the latest versions of both Apps.

See graphic attached.

After designing your route on flyxc.app simply copy the URL and paste into PG-Race or XC Guide.

Alternatively scan the flyxc.app link share QR code into PG-Race or XC Guide (harder to do on the same phone).

PG-Race / XC Guide will convert the route, a simple list of waypoints, into a complete task.

The race task parameters which are currently missing from flyxc.app are automatically inserted using your default values:
SSS turnpoint
ESS turnpoint
SSS start time
Land-by time
Turnpoint radii
Enter/Exit of SSS
Goal Circle or Line
Unfortunately it cannot insert the turnpoint altitudes, which should be present in the task definition. Flyxc does not appear to supply these currently.

Also note that flyxc seems to show distance between waypoint centres, whereas PG-Race / XC Guide will show both distance between centres and optimised.

Be careful with triangles for UK league flights as first and closing TPs must have IDENTICAL centre locations.

There is nowhere in an IGC C-record (declaration header) to indicate turnpoint radius. but I believe radii are assumed to be 400m for UK league flights.

Indy
(PG-Race and XC Guide also include task design tools, incl. turnpoint altitudes, but the flyxc.app UI is much nicer.)

Posted: Thu Jun 13, 2024 0:51 UTC    Post subject: XC Guide altitude correction from AGL or QNH
Another feature of the recent release of XC Guide, which probably deserves some explanation, is the altitude calibration.

Altitude (GPS or Barometric) can be set, whilst on the ground, so that AGL = 0.

QNH can be used for the Barometric altitude correction at any time (including whilst flying).

To set AGL = 0, the AGL (Terrain) feature must be enabled (Flight Computer settings).

A long-press on the altitude box of the Flight Computer offers up to three options:-

1) Set altitude correction so that AGL = 0
2) Set QNH (in mb)
3) Clear the altitude correction

Note: To find out QNH for any known waypoint, XC Guide has the Meteo function, which interrogates https://openweathermap.org/

See attached slide for example use.
